# ADR-0030 — 체크인 후 예약 변경 (연장·빌라이동·인원·추가청구)

- 상태: **Accepted** (테오 확정 2026-07-01 — D1~D3 합의, 설계 착수)
- 관련: ADR-0014(요율 기간), ADR-0018(정산 LEDGER), ADR-0019(체크인·부가), ADR-0021(공급자 직접판매·seller), ADR-0022(파트너 채권), ADR-0027(청구서 수납 LEDGER)
- 코드: `lib/booking-modify.ts`, `lib/availability.ts`, `lib/pricing.ts`, `lib/checkout.ts`, `app/api/bookings/[id]/modify`, `app/(admin)/bookings/[id]/booking-modify-panel.tsx`
- 메모리: [[three-features-revenue-minibar-bookingmod]], [[partner-b2b-receivables-plan]], [[adr-0027-invoice-ledger-collection]]

## 1. 배경 (테오 지적, 2026-07-01)

현재 예약 변경(`lib/booking-modify.ts`, PR #74·#77)은 **상태 무관하게 "전체 재견적"** 방식이다. 확정(CONFIRMED)엔 맞지만 **체크인(CHECKED_IN)엔 틀리다.** 실제 업무에서 체크인 후 다음이 빈번하다:

- 게스트가 **묵는 중에 숙박을 연장**한다 (체크아웃일 추가).
- 그 날 **같은 빌라가 이미 예약**돼서 **다른 빌라로 옮겨** 추가 숙박한다.
- **인원이 늘어난다** (정원 초과 시 더 큰 빌라 필요).
- 빌라를 통째로 **바꾼다**.

현행의 구멍 (코드 확인):
1. **체크인 후엔 체크아웃일만 변경 가능** — 빌라·인원 변경이 `CHECKED_IN_FIELD_LOCKED`로 막힘 (`booking-modify.ts:143`). **인원 변경 잠금은 유지가 맞다(D0 — 리스크).** 빌라 이동·연장만 요구와 불일치.
2. **정원 검증 부재** — `Villa.maxGuests`(schema:141)·침실 `capacity`는 있는데 `checkAvailability`가 안 읽음. 인원이 정원을 넘는지 판정하는 코드가 **어디에도 없다.** (신규 예약·확정 변경·연결 예약의 빌라 선택 시 필요 — 체크인 후 인원 변경은 D0로 애초에 없음)
3. **추가비용 미리보기 부재** — `booking-modify-panel.tsx`는 값만 보내고 토스트만. "이렇게 바꾸면 얼마 추가"를 저장 전에 못 본다.
4. **분할 숙박 불가** — 1 예약 = 1 빌라 = 1 연속 구간이라 "1~3박 A빌라, 4~5박 B빌라"를 담을 수 없다.
5. **단축 시 감액됨** — 전체 재견적이라 날짜를 줄이면 총액이 내려간다. 이미 묵은 밤은 환불 대상이 아닌데도.
6. **과수납 가드 부재** — 싼 빌라로 바꿔 `이미 받은 돈 > 새 총액`이 돼도 경고 없음.
7. **파트너 채권 존재 시 변경 전면 거부** (`booking-modify.ts:249`) — 추가청구 경로 없음.

## 2. 확정 결정 (테오, 2026-07-01)

| # | 질문 | 결정 |
|---|---|---|
| D0 | 체크인 후 인원(guestCount) 변경 | **불허 — 제외.** 리스크가 크다(정원·동의서·보증금·책임이 체크인 시점 인원 기준으로 확정됨). 인원은 **체크인 시점에 잠금**(현행 코드 유지). 인원이 늘면 별도 예약/빌라 추가로 처리 |
| D1 | 분할 숙박(연장인데 원 빌라 불가 → 다른 빌라) 데이터 구조 | **연결된 추가 예약** — 추가 밤을 다른 빌라의 **새 Booking**으로 만들고 원 예약과 `parentBookingId`로 연결. 각 예약은 단일 빌라 유지. 세그먼트 테이블 신설 안 함(대공사 회피) |
| D2 | 체크인 후 단축·싼 빌라 변경 시 금액 | **감액 없음** — 이미 체크인했으면 **최초 확정액을 하한(floor)으로 유지**. 단축·다운그레이드해도 환불 없음. 추가분만 델타로 청구 |
| D3 | 파트너 여신 예약에 추가비용 발생 시 | **청구서 추가라인** — 기존 채권/청구서에 추가 숙박분을 라인으로 더함. 취소·재예약 안 함 |

## 3. 핵심 원칙 적용 (사업 4대 원칙)

- **재고 비공개(원칙1)**: 대체 빌라 후보 제시는 **ADMIN 변경 화면에서만.** 공급자·게스트에 전체 재고 노출 없음.
- **마진 비공개(원칙2)**: 추가청구 미리보기의 판매가·마진은 `canViewFinance` 게이트. 공급자 알림엔 날짜·인원만(금액 값 미포함, 기존 규칙 유지).
- **검수 게이트(원칙3)**: 빌라 이동/추가 대상 빌라는 **반드시 `isSellable`(청소 검수 통과)** 여야 함. 미통과 빌라로는 이동 불가.
- **베트남 UX(원칙4)**: 추가/이동 대상 빌라의 새 공급자에게 vi Zalo 알림 + 청소 태스크.

## 4. 설계 — 상태별로 계산 방식이 갈린다 (핵심)

```
확정(CONFIRMED) — 아무도 안 묵음
  → 전체 재견적 OK (올려도 내려도 됨). 단축 감액 정상.
  → 단, 싼 빌라 변경으로 과수납되면 경고 (E절)

체크인(CHECKED_IN) — 이미 묵는 중
  → 지난 밤 잠금(재계산·환불 금지). 미래분만 변경 가능.
  → 인원(guestCount)은 잠금 — 변경 불가 (D0)
  → 청구액 = 최초 확정액 + Σ(추가된 밤·빌라 요금)   ← 델타 방식, 하한=최초액
  → 단축·다운그레이드해도 감액 없음 (D2)
```

`modifiableKind`를 확장: 체크인 후에도 **체크아웃 연장·빌라이동**을 허용하되(계산은 델타), **인원은 계속 잠금(D0).**

## 5. 시나리오별 처리 (체크인 후)

| 시나리오 | 처리 |
|---|---|
| **연장 · 같은 빌라 공실** | 체크아웃일 뒤로. 추가 밤 요금만 델타 청구 (기존 예약 update) |
| **연장 · 원 빌라 불가 → 다른 빌라** | **D1: 새 Booking(대체 빌라·추가 구간) 생성 + `parentBookingId` 연결.** 대체 빌라는 공실+정원+`isSellable` 충족만 후보. 새 빌라 공급자 알림·청소 태스크 |
| ~~인원 추가~~ | **D0로 제외.** 체크인 후 인원 변경 불허(리스크). 실제로 인원이 늘면 별도 예약/빌라 추가로 처리 |
| **빌라 전체 변경** | 새 빌라 공실+정원+`isSellable`. 미래분만 재계산(지난 밤은 원 빌라 요금 유지). 원 빌라 조기 체크아웃 청소 + 새 빌라 준비. 인원은 불변(D0) |
| **단축** | 최초액 유지(감액 없음). 실제 조기 퇴실이면 원 빌라 청소 앞당김. 요금 불변 |

## 6. 데이터 모델

> 스키마 변경은 라이브 드리프트 위험으로 `prisma db push` 금지 — additive 컬럼은 raw SQL `ALTER` ([[db-schema-drift-villa-source]]).

- `Booking.parentBookingId String?` (additive, self-relation) — 연결된 추가 숙박 예약이 원 예약을 가리킴. null=독립 예약. 원 예약 상세에 "추가 숙박" 카드로 자식 예약 노출.
- 파트너 채권(D3): 추가 예약(자식)의 금액을 **부모의 `PartnerReceivable`/청구서에 라인 추가**로 합산. 자식은 별도 채권을 만들지 않음(이중 인식 방지 — ADR-0027 패턴).
- 델타 청구(D2): 별도 테이블 불필요. 체크인 후 변경은 `Booking.totalSale*`를 **`max(기존액, 재견적액)` 규칙**으로 갱신. 추가금은 기존 `Payment` 흐름으로 수납.

## 7. UI (신규 필요)

1. **변경 미리보기(dry-run) 견적 엔드포인트** — 빌라·날짜·인원 선택 시 저장 전에 `{추가청구, 새 총액, 정원 OK/초과, 공실 OK/불가, 과수납 여부}` 반환. `POST /api/bookings/[id]/modify/preview` (커밋 없음).
2. **정원·공실 필터 빌라 셀렉터** — 구간·인원 넣으면 **공실 + 정원충족 + isSellable** 빌라만 노출.
3. **연장 시 "다른 빌라 추가" 흐름** — 원 빌라 불가하면 대체 후보 제시 → 추가 예약으로 연결.

## 8. 과수납/환불 가드 (E)

변경 저장 전 `이미 수납액(Payment 합계) vs 새 예상액` 비교 → `수납 > 새 총액`이면 "과수납 X — 환불 필요" 경고 + 확인 요구. 체크인 후엔 D2(감액 없음)로 대부분 예방되나, **확정 상태 싼 빌라 변경**에서 발생 가능.

## 10. 작업 분해 (TASKS 후보 — 방향 무관 선행부터)

- **T-A (선행·독립)**: `checkAvailability`에 정원 검증 옵션 추가 — 신규 예약·확정 변경·연결 예약의 **빌라 선택**에 적용(정원 초과 빌라 거부). 체크인 후 인원 변경은 D0로 없음.
- **T-B (선행·독립)**: 변경 미리보기 dry-run 엔드포인트 + 패널 "추가청구/새 총액" 표시.
- **T-C**: 상태별 재계산 분기 — 체크인 후 델타·하한(max) 규칙 (D2). 단축 감액 차단 + 테스트.
- **T-D**: 과수납 가드 (E) — 확정 다운그레이드 경고.
- **T-E (대형)**: 분할 숙박 = `parentBookingId` + 연결 추가 예약 생성 흐름 (D1). 대체 빌라 후보·새 공급자 알림·청소.
- **T-F**: 파트너 채권 추가라인 (D3) — 자식 예약 금액을 부모 청구서에 합산.
- **T-G**: 게스트/공급자/청소 후속 반영 (출입정보·정산 미리보기·알림).

권장 순서: T-A·T-B(즉시 가치, 저위험) → T-C·T-D → T-E·T-F(대형) → T-G.
