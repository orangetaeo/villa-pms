# 기획 — 공급자 계약 「취소 수수료 5단계표」 도입 + 협의(네고) 루프

관련: `docs/business/contracts/01-villa-supply-agreement.md`(내부 초안), `docs/business/contracts/signing/villa-supply.{ko,vi}.md`(서명 정본), `lib/business-contract.ts`, `lib/cancellation-policy.ts`

## 1. 요구 (테오)

체크인 시점별 **고객 환불률 / 공급자 지급률** 5단계표를 빌라 공급자 계약에 넣어 **제안**하고, 공급자가 **우리와 협의(역제안)** 할 수 있게 한다.

| 취소 접수 시점 | 고객 환불 | 공급자 지급 |
|---|---|---|
| 체크인 14일 전까지 | 결제금액의 100% | 공급가의 0% |
| 체크인 8~13일 전 | 80% | 20% |
| 체크인 1~7일 전 | 50% | 50% |
| 체크인 당일 | 20% | 80% |
| 노쇼 또는 체크인 후 취소 | 환불 없음 | 100% |

---

## 2. 현황 진단 — 취소 정책이 3벌, 서로 기준이 다름

| 층위 | 현재 값 | 저장 |
|---|---|---|
| ① 고객 환불 | 30일 전 100% / 14일 전 50% / 이후 0% | AppSetting `CANCELLATION_POLICY` (3단계 고정 구조) |
| ② 공급자 지급 | 14일 전 0% / 이후 원가 50% / 당일·노쇼 100% | 계약 `termsJson.cancelFreeDays`(14)·`cancelPartialPct`(50) |
| ③ 제안표 | 5단계, 양축 동시 | **없음** |

**문제:** ①과 ②가 back-to-back이 아니다. 예) 고객이 체크인 20일 전 취소 → 고객은 100% 환불(우리 수입 0), 공급자 지급도 0% → 무손실 OK. 그러나 **체크인 25일 전** 취소는 고객 100% 환불인데 공급자 별표2 기준으로도 0% → OK. 반대로 **고객 정책 14일 이내 = 50% 환불**인데 공급자에겐 **원가 50%** 지급 → 판매가 100만/원가 70만이면 우리 수취 50만 − 지급 35만 = **15만(마진의 50%)** 확보. 즉 현재도 우연히 비례에 가깝지만, 두 표의 **일수 경계(30/14 vs 14/8/1)가 달라 구간이 어긋나는 지점**에서 손실이 난다. 5단계표는 이 어긋남을 없애는 것이 본질.

---

## 3. 제안표의 구조적 해석 — 왜 이 표가 옳은가 (근거 계산)

각 단계에서 **고객 위약금률 = 공급자 지급률**(100−환불률 = 지급률)로 설계돼 있다. 이 경우 우리 몫은 항상 **마진 × 위약금률**이 되어, 어떤 시점에 취소돼도 **회사는 절대 손실이 나지 않고, 손실을 공급자와 정확히 같은 비율로 나눈다.**

예: 판매가 KRW 1,000,000 / 공급가 KRW 700,000 환산(마진 300,000)

| 시점 | 고객에게서 수취 | 공급자에게 지급 | 회사 몫 | 마진 대비 |
|---|---|---|---|---|
| 14일 전 | 0 | 0 | 0 | 0% |
| 8~13일 | 200,000 | 140,000 | 60,000 | 20% |
| 1~7일 | 500,000 | 350,000 | 150,000 | 50% |
| 당일 | 800,000 | 560,000 | 240,000 | 80% |
| 노쇼 | 1,000,000 | 700,000 | 300,000 | 100% |

→ **협상 논리로 그대로 쓸 수 있다**: "우리는 위약금을 독식하지 않는다. 같은 비율로 나눈다." 베트남어 제안서 문구도 이 한 줄이 핵심.

**단, 이 비례는 아래 4개 조건이 지켜질 때만 성립한다.** 하나라도 깨지면 회사가 손실을 뒤집어쓴다 → §4의 미결 쟁점.

## 4. 반드시 먼저 결정할 쟁점 (테오 판단 — 코드보다 선행)

### 쟁점 1 ★ "결제금액의 %"의 기준 — 부분입금 함정
선금 30%만 받은 예약이 8~13일 전 취소되면?
- (A) 문자 그대로 "결제금액(=선금 300,000)의 80% 환불" → 우리 수취 60,000. 그런데 공급자에겐 **공급가의 20% = 140,000** 지급 → **회사 순손실 80,000.** ❌
- (B) **총 예약금액 기준 위약금**(1,000,000×20% = 200,000)을 산정 후 결제액에서 차감 → 환불 100,000, 수취 200,000, 지급 140,000, 회사 +60,000. ✅

→ **권고: (B). 계약·고객 약관 문구를 "총 예약금액 기준"으로 확정하고, 위약금이 기수령액을 초과하면 잔여분을 청구(미청구 시 회사 부담)라고 명시.** 이 한 줄이 빠지면 표 전체가 손실 구조가 된다.

### 쟁점 2 환차(KRW 수취 / VND 지급)
고객 위약금은 KRW, 공급자 지급은 VND. 환율 변동 부담자 미정.
→ **권고: "공급자 지급액은 해당 예약의 원가(VND) × 지급률로 확정하며, 환율 변동은 회사가 부담"** (공급자에겐 VND 고정이라 단순 = 베트남 UX 원칙 부합, 회사 리스크는 마진 내에서 흡수).

### 쟁점 3 부가서비스·티켓 포함 여부
BBQ·입장권·차량 등 벤더 발주분은 별도 취소정책(벤더별)이 있고 이미 발주된 건은 회수 불가.
→ **권고: 5단계표는 숙박료에만 적용. 부가서비스는 "발주 완료분은 취소 시 전액 고객 부담" 별도 조항.** 계약 별표2 상단에 적용 범위 1줄 명시.

### 쟁점 4 판매 경로별 예외
- 파트너(B2B) 등급별 정책: 이미 "14일 내 = 홀드 없음 100%" 규칙 존재 → 5단계표와 충돌 없는지 확인 필요.
- 공급자 직접판매(seller=SUPPLIER, 공급자 100% 수금): **적용 대상 아님** → 별표2에 제외 명시.

### 쟁점 5 고객 정책도 5단계로 바꿀 것인가
현재 고객 정책은 30/14 3단계. 표를 그대로 쓰면 고객 무료취소가 **30일 → 14일로 축소**(고객에게 불리) + 단계가 5개로 증가.
→ **권고: 예. 안 바꾸면 back-to-back이 깨진다.** 다만 이는 **고객 대면 약관 변경**이므로 이미 동의 스냅샷(`policyConsentJson`)이 저장된 기존 예약에는 소급 금지(스냅샷 정본 원칙 그대로 유지).

## 5. 설계

### 5.1 데이터 — 계약 termsJson (additive, 스키마 변경 없음)

`lib/business-contract.ts` `villaSupplyTermsSchema`에 티어 배열 추가. 기존 2필드는 **레거시 호환용으로 남기고**(서명 완료 계약 렌더 보존) 신규 계약은 티어 배열 사용.

```ts
cancelTiers: z.array(z.object({
  fromDays: z.number().int().min(-1).max(365),  // 구간 시작(체크인 기준 D-n). -1 = 노쇼/체크인 후
  toDays:   z.number().int().min(-1).max(365),  // 구간 끝
  guestRefundPct:  z.number().int().min(0).max(100), // 고객 환불률(참고 표시)
  supplierPayPct:  z.number().int().min(0).max(100), // ★ 계약의 구속력 있는 값
})).min(1).max(8).optional()
```

- **정합성 검증(서버)**: 구간 연속·중복 없음, 일수 내림차순, `supplierPayPct ≤ 100 − guestRefundPct`(회사 손실 방지 상한). 위반 시 400 + 관리자 폼에 경고.
- ★ 마진 비공개 원칙 유지: 금액이 아닌 **비율만** 저장. 판매가·마진 필드는 여전히 금지(.strict 유지).

### 5.2 서명 정본 (ko/vi) — 표준버전 v0.4 → **v0.5**

`signing/villa-supply.ko.md` · `.vi.md` 별표 2를 5행 표로 교체. 토큰은 행 반복이 필요하므로 **단일 토큰 `{{cancelTiersTable}}`로 표 전체를 서버에서 마크다운 문자열로 렌더 주입**(현재 `{{token}}` 치환기 구조 그대로 사용 가능, 미치환 검사도 유지).

별표2에 §4 결정사항 3줄 추가:
1. 적용 범위: 숙박료. 부가서비스 발주분 제외.
2. 기준: 총 예약금액 기준 산정(부분입금 무관).
3. 지급액은 원가(VND) 고정, 환율 변동은 회사 부담.

> ★ 정본 수정 시 규칙(기존): 내부 초안(`01-villa-supply-agreement.md`) 동시 갱신 + `협상 여지|변호사 확인` grep 0 확인.

### 5.3 고객 정책 확장 (`lib/cancellation-policy.ts`)

현재 `{fullDays, partialDays, partialPct, enabled}` 고정 3단계 → **`tiers: []` 배열 v2로 확장, v1 JSON은 파싱 시 자동 승격**(하위호환). 공개 페이지·동의 스냅샷은 배열을 그대로 렌더. `policyConsentJson`은 스냅샷이므로 기존 예약 영향 없음.

### 5.4 ★ 협의(네고) 루프 — 3안 비교

| 안 | 내용 | 장점 | 단점 |
|---|---|---|---|
| A. 오프라인 | Zalo 채팅으로만 협의 | 코드 0 | 이력·증빙 없음, 어느 조항인지 불명 |
| B. 상태 확장 | `BusinessContractStatus`에 `NEGOTIATING` 추가 | 상태 명확 | enum ALTER TYPE + 전 switch 영향 |
| **C. 협의 레코드(추천)** | 신규 테이블 1개 + 파생 차단 | additive·롤백 쉬움·조항 단위 이력 | 상태 표시가 파생 계산 |

**추천 C안 상세**

```prisma
model ContractNegotiation {
  id         String   @id @default(cuid())
  contractId String                  // BusinessContract.id (관계 미설정, id 보존 — 기존 관례)
  clauseKey  String                  // "cancelTiers" | "payMethod" | "specialTerms" ...
  reason     String                  // 프리셋 코드(자유서술 최소화 — VN UX)
  proposed   Json?                   // 역제안 값(예: 티어 배열)
  note       String?  @db.Text       // 선택 자유 메모(vi)
  status     String   @default("OPEN") // OPEN | ACCEPTED | REJECTED
  resolvedById String?
  resolvedNote String?
  createdAt  DateTime @default(now())
  resolvedAt DateTime?
  @@index([contractId, status])
}
```

흐름:
1. 포털(공급자) 계약 열람 화면 — 각 별표 옆 **「협의 요청」 버튼**. 베트남 공급자 UX 원칙상 **텍스트 입력 강요 금지**: 프리셋 사유 칩(예: "무료취소 기간이 너무 김" / "당일 지급률이 낮음" / "성수기는 다른 조건 원함") + 숫자 역제안은 **드롭다운/스텝퍼**로.
2. 저장 → **운영자 Zalo 통지**. ★ 신규 `NotificationType` 추가는 enum switch 전수 영향이 있으므로 **가급적 기존 타입 재사용 + payload 분기**를 먼저 검토(불가피하면 `ALTER TYPE ADD VALUE` + `docs/NOTIFICATIONS.md` 동시 갱신).
3. `status=SENT`이고 **미해결(OPEN) 협의가 있으면 포털 서명 버튼 비활성**("협의 진행 중") — enum 변경 없이 파생 판정.
4. 운영자 `/contracts/[id]` 협의 탭에서 **수용**(→ termsJson 수정 + 계약 재발송, 감사로그) 또는 **거절**(사유 입력, 공급자에게 통지) → OPEN 해소 → 서명 가능.
5. 서명 시 `contentHash` 봉인은 현행 그대로 — 협의 이력은 별도 테이블로 남아 분쟁 증빙이 된다.

### 5.5 "제안" 단계 산출물 (계약 이전)

계약 서명 요청 전에 보여줄 **1페이지 조건 요약(vi/ko)** — 5단계표 + §3 "손실을 같은 비율로 나눈다" 논리 + 우리 의무(선입금·정산주기). 기존 `/documents` 내부 뷰어 재사용, 인쇄용은 기존 헤드리스 print-PDF 기법 사용.

| 리스크 | 완화 |
|---|---|
| 고객 무료취소 30→14일 축소 = 전환율 하락 | 마케팅 문구 조정, A/B 없이 즉시 전환 시 문의 증가 대비. 쟁점 5에서 "고객 100% 구간만 30일 유지, 공급자 0% 구간도 30일로" 대안 검토 가능(단 8~13/1~7 구간은 그대로) |

## 8. 즉시 필요한 테오 결정 (5줄)

1. 위약금 기준 = **총 예약금액**으로 확정? (권고 예)
2. 환율 변동 = **회사 부담**으로 확정? (권고 예)
3. 부가서비스 = **표 적용 제외**? (권고 예)
4. 고객 정책도 **5단계로 교체**? 아니면 무료취소만 30일 유지?
5. 협의 루프 = **C안(협의 레코드)** 채택?
