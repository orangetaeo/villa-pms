## 범위 (In)

1. **lib/proposal.ts** — 제안 생성 규칙 단일 소스
   - 순수 함수 층 (단위 테스트 대상):
     - `defaultCurrencyForChannel(channel)` — DIRECT→KRW, TRAVEL_AGENCY·LAND_AGENCY→VND (ADR-0003, ADMIN 오버라이드는 입력으로)
     - `effectiveProposalStatus(status, expiresAt, now)` — ACTIVE인데 expiresAt 경과면 EXPIRED 판정 (c2 "서버 판정값" — cron 없이 시각 기준, evaluateProposalForHold와 동일 규약)
     - `uniformNightlyPrice(nightly[])` — 박별 요율이 전부 동일하면 그 값, 시즌 경계로 섞이면 null (ProposalItem.price*PerNight은 균일가일 때만 — 평균 등 가공 금지, money-pattern)
     - `generateProposalToken()` — crypto.randomBytes(24) base64url. 스키마 default cuid()는 추측 가능성이 있어 공개 링크 토큰으로 부적합 — 생성 시 명시 주입
   - DB 층: `createProposal(prisma, input)` — 트랜잭션: 항목별 ① checkAvailability — sellable 아니면 거부(빌라명·사유 반환) ② quoteStayForVilla로 saleCurrency 요율 박별 합산 ③ assertSaleAmountColumns 검증 → Proposal+items 생성, fx 스냅샷(getFxVndPerKrw, 미설정 null), AuditLog
2. **app/api/proposals/route.ts** — POST 생성 + GET 목록 (둘 다 첫 줄 ADMIN 검사)
   - POST 입력(zod): clientName, channel, saleCurrency?(기본값=채널 매핑), expiresInHours?(기본 48, 1~336), note?, items[]: {villaId, checkIn, checkOut} — 날짜는 `parseUtcDateOnly` 정규화(T1.3 QA 권고), 항목 1~3개·빌라 중복 금지
   - GET 목록: effectiveStatus 포함, b12용 기본 필드 (serializeBigInt)
3. **app/api/proposals/candidates/route.ts** — GET ?checkIn&checkOut&saleCurrency: `findSellableVillaIds`(villaIds 생략 = 전체 재고 — **ADMIN 전용 강제**, T1.3 QA 권고·leak-checklist) → 빌라 정보+견적(원가 포함 — ADMIN이므로 허용). 요율 미설정(MissingRateError) 빌라는 후보 제외 + warnings 목록 반환
4. **단위 테스트** (lib/proposal.test.ts) — 순수 함수 층 전수

## 스코프 결정 (PM/TDA 이관 사항의 본 계약 처리)

- **holdHours 영속(T2.3 QA 이관 ②)**: Phase 1은 **(b) AppSetting `HOLD_HOURS_DEFAULT` 전역값만** 채택 — 제안별 24/48h 선택은 Proposal.holdHours 스키마 추가가 필요해 규칙 #6(스키마 변경 전담 세션) 위반 위험. POST 입력에서 holdHours를 받지 않고, 전역값은 T1.7-FE /settings에서 관리. 제안별 선택은 TASKS 잔여로 유지(TDA 스키마 검토 시 createHold의 override 인자가 이미 수용)
