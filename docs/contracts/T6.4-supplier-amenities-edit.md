# T6.4(잔여) — 공급자 빌라 비품 수정

> 담당: UX-VN/BE · 합의: QA · 상태: 착수 (2026-06-16)

## 1. 배경·스코프 확정

T6.4 비품의 잔여 2건 중:
- **ADMIN b10 비품 조회 = 이미 완료** (app/(admin)/villas/[id]/page.tsx 읽기 전용 2x2 + 미니바 렌더 확인).
- **잔여 = 공급자 내 빌라 상세 비품 수정** — 등록(T1.1) 후 비품을 마법사 반려-재제출 없이 직접 갱신.
  현재 공급자는 등록 시점에만 비품 입력 가능, 이후 수정 경로 없음(REJECTED 재제출 PUT은 전체 입력 재수신).

## 2. 범위 (신규 위주 — 충돌 0)

- **신규**: `app/api/villas/[id]/amenities/route.ts` — PATCH(SUPPLIER, supplierId 스코프). 비품 전체 교체

- **신규**: `app/(supplier)/my-villas/[id]/amenities/amenities-editor.tsx` — 클라(a9 UI 패턴 재사용: 카테고리 4탭·체크 타일·미니바 스테퍼) + 저장 버튼 → PATCH → 상세로 복귀.
- **수정(본인 소유)**: `app/(supplier)/my-villas/[id]/page.tsx` — 비품 카드에 "Sửa"(수정) 진입 링크.

## 3. 규칙

- **누수 불변**: 비품은 판매가·마진과 무관 — 이 작업은 VillaRate를 일절 조회/수정하지 않음.
- **상태 불변**: 비품은 승인 게이트 대상이 아님(Phase 1 표기·조회용) → 수정해도 villa.status 변경 없음(재승인 불필요).
- **소유 검증**: 타인·부재 빌라 동일 404(notFound), 비SUPPLIER redirect. PATCH는 supplierId 스코프 updateMany식 가드.
- itemKey는 lib/amenities 사전 값만(isValidAmenity), 미니바만 수량>1 의미, quantity 1~99.

## 4. 완료 기준 (테스트 가능)

1. 상세 비품 카드 "Sửa" → `/my-villas/[id]/amenities`: 현재 비품 prefill, 카테고리 탭·체크·미니바 수량 편집.
2. 저장 → PATCH 200 → 상세로 복귀, 변경 반영. 빈 비품(전체 해제) 저장 가능.
3. PATCH 권한: 비로그인 401, 비SUPPLIER 403, **타인 빌라 404**(존재 비노출), 잘못된 itemKey 400.
4. AuditLog 기록(entity=Villa, action=UPDATE, changes에 비품 개수 전후) — 변경 추적.
5. 누수: 라우트·페이지가 salePrice·margin·VillaRate 미참조(grep 실증).
6. i18n ko/vi 동기(amenities.save 등), 하드코딩 0. `npx tsc --noEmit`·`npx vitest run` 신규 에러 0.
7. tests/villa-amenities-api.test.ts — 검증 로직(isValidAmenity·권한 분기·payload 파싱) 단위 테스트.

## 5. 비고
- 비품 편집은 모든 소유 빌라 상태에서 허용(REJECTED 재제출 마법사와 별개의 빠른 수정 경로).
- 디자인: 신규 Stitch 없이 a9-amenities(등록 마법사 4/5) UI 패턴 재사용 — 동일 컴포넌트 톤, 푸터만 저장/취소로 교체.
