## 범위 (In)

### A. 스키마 + ADMIN 반려
1. **schema 3건 + db push** (상기 TDA 결정)
2. **PATCH /api/villas/[id] REJECT 액션** — body `{ action: "REJECT", reason: string(1~1000 trim) }`. 전이 PENDING_REVIEW→REJECTED(가드 트랜잭션, 409). Villa.rejectionReason 저장, writeAuditLog(status·rejectionReason changes), Notification(VILLA_REJECTED → villa.supplierId, payload에 villaId·villaName·reason — 마진·금액 미포함). REJECT는 초기 검수 태스크 미생성(APPROVE 전용 유지)
3. **ADMIN 반려 모달** (b10 디자인) — `villa-actions.tsx`의 PENDING_REVIEW 분기에 "반려" 버튼 추가 + 반려 사유 textarea 모달(사유 미입력 시 확정 disabled). 별도 client 컴포넌트 분리 가능. 성공 시 refresh

### B. 공급자 가시화 + 재제출
4. **`/my-villas` 목록** (a6 변환, RSC + prisma 직접 조회, supplierId 스코프) — 빌라 카드(대표 사진·이름·단지·상태 배지 5종 DRAFT/PENDING_REVIEW/REJECTED/ACTIVE/INACTIVE). **REJECTED 카드는 빨강 배지 + 반려 사유 한 줄**. 카드 클릭 → 상세. 기존 "+ Thêm villa" FAB 유지. 빈 목록 안내. **마진·판매가·KRW 절대 미조회**(SUPPLIER 화면 — select에 원가만)
5. **`/my-villas/[id]` 상세** (a10 변환, RSC, supplierId 스코프 — 타인 빌라 404) — 빌라 정보(사진·기본정보·비품·시즌 원가). **REJECTED일 때 반려 블록**(사유 + "Sửa và gửi lại" 버튼 → `/my-villas/[id]/edit`). PENDING_REVIEW·ACTIVE 등은 상태 배지만
6. **재제출 = 마법사 edit mode**:
   - `villaToWizardState(villa)` 순수 함수(wizard-types.ts) — Villa+photos+amenities+rates → WizardState. 사진은 space+spaceLabel→슬롯 id 매핑(done 상태), 원가는 supplierCostVnd 문자열
   - `villa-wizard.tsx`에 `initialState?`·`villaId?` prop — villaId 있으면 PUT, 없으면 POST(기존)
   - `/my-villas/[id]/edit/page.tsx`(RSC, supplierId 스코프 + **REJECTED 상태만 허용**, 그 외 상세로 redirect) → villa 로드 → 변환 → `<VillaWizard initialState villaId>`
   - **PUT /api/villas/[id]** — SUPPLIER 전용 + supplierId 스코프 + **REJECTED 상태만**(409). villaCreateSchema 재사용. 트랜잭션: villa.update(기본정보) + photo deleteMany+createMany + amenity deleteMany+createMany + rate deleteMany+createMany(POST와 동일 — marginValue=0·salePriceVnd=cost 리셋, ADMIN 미승인 상태라 안전) + status PENDING_REVIEW + rejectionReason null. writeAuditLog
7. **단위 테스트**: villaToWizardState(사진 슬롯 매핑·원가·비품 키), REJECT API(401/403/404/409/사유 필수 400/AuditLog), PUT API(403 타인·409 비REJECTED·성공 시 전이+rejectionReason null), messages 키
