# 계약: F10 Phase B — 공급자 판매가 + 공개 판매 링크 (T10.6·T10.7, ADR-0021 D1)

날짜: 2026-06-26 / 담당: TDA·BE·UX-VN·FE / 평가: QA(독립)
브랜치: `wt/f10-phase-b` (origin/main = PR #48 머지 `fada02b` 기준 — **Phase A 인프라 포함**)

**착수 선점 선언(이 계약 단독 커밋+푸시). Phase A(PR #48, seller=SUPPLIER·supplierSalePriceVnd·검수)는 전제·완료.**

## 배경

Phase A는 공급자가 오프라인 판매를 **수동 기록**(캘린더 직접예약)하는 단방향이었다. Phase B는 공급자가 **자기 고객에게 공개 판매 링크를 발급** → 고객 셀프 가예약(HOLD) → 입금 확인 → CONFIRMED 까지 자동화한다. 이로써 "공급자 직접 판매 도구 = 공실 공유 엔진"이 완성된다(ADR-0021 §1).

핵심 재사용: **기존 Proposal/`/p/[token]` 가예약 파이프라인(F3)**과 **VillaRatePeriod 자동 견적(ADR-0014, [[rate-periods-status]])**을 `seller=SUPPLIER`·supplierId 스코프로 분기. 신규 판매 인프라 작성 금지 — 기존 경로 분기.

## 절대 불변식 (사업 4대 원칙 — 위반 시 즉시 실패)

- **재고 비공개(원칙1)**: 판매 링크·셀프 가예약 화면은 **그 공급자 자기 빌라만** 노출. 타 공급자 빌라·전체 재고 도달 0건.
- **마진 비공개(원칙2)**: 공급자 화면·판매 링크·견적 어디에도 운영자 `salePriceKrw`·마진·미니바 `costVnd` 노출 0건. 노출 금액은 공급자 소유 `supplierSalePriceVnd`(VND)만.
- **선착순(D2)**: 셀프 가예약도 **기존 `lockVillaInventory()`+`checkAvailability()` 트랜잭션** 통과 — 우선권 로직 0줄. 운영자가 먼저 잡은 날짜면 HOLD 실패(상세 비노출).
- **정산 제외(D3)**: `seller=SUPPLIER` 예약은 정산 집계에서 제외(Phase A 필터 계승).
- **베트남 UX(원칙4)**: 공급자 판매가 입력·링크 발급은 vi 기본·모바일·텍스트 최소.

## ① 구현 범위

### T10.6 — 공급자 판매가 (TDA·UX-VN)
- **스키마**: `VillaRatePeriod.supplierSalePriceVnd BigInt?` 추가. **`prisma db push` 금지** — raw SQL `ALTER TABLE ... ADD COLUMN`(additive, [[db-schema-drift-villa-source]]) + `prisma generate`. TDA 전담.
- **입력 UI**: `app/(supplier)/my-villas/[id]/rate-periods`(또는 기존 요율 편집 화면)에 기간별 `supplierSalePriceVnd` 입력칸 추가. vi·모바일. 공급자 소유 정보이므로 원가(`supplierCostVnd`)와 나란히 표시 가능하나 운영자 `salePriceVnd`/마진은 비노출.
- **권한**: `villa.supplierId === session.user.id` 스코프 강제.

### T10.7 — 공급자 공개 판매 링크 (BE·FE·UX-VN·QA)
- **권한 분리**: 신규 `canCreateSupplierLink`(공급자가 자기 빌라 판매 링크 생성). 운영자 제안링크 생성 권한과 분리 — 공급자는 자기 빌라만.
- **링크 생성**(공급자, vi): 기존 Proposal 엔티티 재사용 + `seller=SUPPLIER`·생성자 supplierId 스코프. 자기 빌라 1개+기간 선택 → 토큰 발급. 견적은 T10.6 `supplierSalePriceVnd` 자동 적용(없으면 입력).
- **셀프 가예약**(공급자 고객, 비로그인): 기존 `/p/[token]` 흐름 재사용. 고객이 날짜·인원·고객명 입력 → `POST` → **기존 가용성 트랜잭션** 통과 시 `Booking(seller=SUPPLIER, status=HOLD)` + 홀드 만료(기존 cron 재사용). 패배 시 "이미 예약됨"(상세 비노출).
- **입금 확인 → CONFIRMED**: 공급자가 입금 확인 시 HOLD→CONFIRMED 전환(자기 `supplierId`·`seller=SUPPLIER` 스코프). 기존 confirm 경로 재사용.
- **Zalo 알림**(INTEG, 선택): 셀프 가예약 발생 시 공급자 Zalo 통지(기존 NotificationType 패턴).

### 범위 밖 (명시)
- Phase A 전체(수동 기록·검수·미니바 정산) — PR #48 완료, 전제.
- 미니바 전환별 자동 보충/회수 — ADR-0019 Phase 2.
- 운영자 마진·재판매가 노출 일절 — 영구 금지.
- 공급자 판매 링크의 다중 빌라 비교(운영자 제안링크 전용) — 공급자는 자기 빌라만.

## ② 테스트 가능한 완료 기준 (QA 하드 게이트)
1. `VillaRatePeriod.supplierSalePriceVnd` raw SQL ALTER 적용·라이브 DB 반영 확인, `prisma generate` 후 typecheck 0.
2. 공급자가 자기 빌라 판매가 입력·저장 → 재조회 일치. 타 공급자 빌라 입력 시 403/스코프 차단.
3. 판매 링크 생성 → `/p/[token]` 셀프 가예약 → `Booking(seller=SUPPLIER, status=HOLD)` 생성. 운영자 선점 날짜는 HOLD 실패.
4. 입금 확인 → CONFIRMED 전환. 타 공급자/운영자 예약은 전환 불가.
5. 정산 집계에서 `seller=SUPPLIER` 제외 유지(회귀 없음).
6. **누수 QA(독립)**: 공급자 화면·판매 링크·견적·API 응답 전 경로 grep — `salePriceKrw`·마진·미니바 `costVnd`·타 공급자 빌라 노출 **0건**. 공격 시나리오(타 supplierId 토큰·직접 API 호출) 차단.
7. `npm run lint && npm run typecheck` 0, 관련 테스트 통과, `next build` 성공([[deploy-build-gate]]).
