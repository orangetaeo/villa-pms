## 배경 (실측으로 드러난 공백)

`/api/villas/[id]/clips`(presign·커밋·PATCH·DELETE)는 **SUPPLIER와 운영자를 모두 허용**한다.
그런데 업로드 **화면**은 공급자 전용 `/my-villas/[id]/videos` 하나뿐이고, 이 경로는
`(supplier)` 레이아웃이라 운영자로 접근하면 `/dashboard`로 리다이렉트된다(2026-07-23 실측).

결과: **테오(OWNER)는 자기 손으로 빌라 영상을 올릴 방법이 없다.** 실제로 오늘 검증에서
운영자 계정으로는 UI 업로드가 불가능해 서버 자격증명·브라우저 콘솔로 우회해야 했다.

운영자 화면(`/villas/[id]`)에는 이미 「영상 클립」 카드(`clip-review.tsx`)가 있고
승인·반려·삭제만 한다. **업로드가 빠진 자리가 정확히 여기다.**

## 범위 (포함)

1. `app/(admin)/villas/[id]/clip-review.tsx`에 **업로드 기능 추가**
   - 3단계 흐름은 공급자와 동일: presign → 브라우저 R2 직접 PUT(진행률) → 커밋(서버 실측 검증)
   - 다크 테마·한국어(운영자 화면 규칙)
2. **운영자에게만 있는 입력**: `space`(공간) 선택 + `note`(메모)
   - 공간은 나레이션·자막 자동 매칭(`captionForPhotoSpace`)에 물리므로 소재 품질에 직결된다.
     공급자 화면은 텍스트 입력 최소화 원칙 때문에 안 받지만, 운영자 화면은 받는 게 맞다.
3. **업로드 전 사전 검사**(모바일 데이터·시간 낭비 방지) — 타입·크기·길이.
   브라우저가 메타데이터를 못 읽으면 통과시키고 **서버 판정에 위임**(과잉 거절 방지).
4. **정책값은 서버에서 받는다** — `GET /api/villas/[id]/clips`의 `policy`
   (AppSetting 오버라이드가 화면에 그대로 반영돼야 한다. 하드코딩 금지)
5. 순수 함수는 `lib/villa-clip-upload.ts`로 분리 + 유닛 테스트
6. i18n ko/vi 동시 추가(`adminVillas.detail.clips` — 기존 NS)

## 제외

- **공급자 컴포넌트(`clip-manager.tsx`) 리팩터링·공용화** — 오늘 R2 CORS 수정으로
  **막 정상 동작하기 시작한 경로**다. 지금 건드리면 회귀 위험이 이득보다 크다.
  중복 로직 정리는 별도 태스크(백로그)로 남긴다.
- 드래그앤드롭·다중 업로드 — 운영자는 소수 파일을 올린다. 단순하게 간다.
- 클립 순서 변경·트리밍 — 편집은 쇼츠 마법사가 담당한다.
