## 범위 (In)

1. **lib/cleaning.ts** — 청소·검수 게이트 단일 소스
   - 순수 함수 층 (단위 테스트 대상):
     - `assertCleaningTransition(current, next)` — 상태기계: PENDING→PHOTOS_SUBMITTED, REJECTED→PHOTOS_SUBMITTED(재업로드), PHOTOS_SUBMITTED→APPROVED|REJECTED. 그 외 전이 거부
     - `canOpenSellableGate(openCheckoutTaskCount)` — **게이트 규칙**: 해당 빌라에 미결(OPEN=PENDING·PHOTOS_SUBMITTED·REJECTED) CHECKOUT 태스크가 0건일 때만 isSellable=true 허용 — PERIODIC 승인이 체크아웃 게이트를 우회하지 못하게 함
     - `monthKeyVn(date)` — 정기 방역 멱등 키 (Asia/Ho_Chi_Minh 기준 YYYY-MM)
   - DB 층 (전부 트랜잭션 + Notification 큐 + AuditLog):
     - `createCheckoutCleaningTask(db, {bookingId, actorUserId, now})` — **DbClient 주입**(T3.3 체크아웃 트랜잭션 안에서 호출): CleaningTask(CHECKOUT, PENDING, dueDate=booking.checkOut) 생성 + **villa.isSellable=false(게이트 닫기)** + Notification(CLEANING_REQUEST → assignee 있으면 청소자, 없으면 공급자 — SPEC CLEANER 운영 방식)
     - `submitCleaningPhotos(prisma, {taskId, photoUrls(≥1), actorUserId})` — →PHOTOS_SUBMITTED (전이 검증·status 가드)
     - `approveCleaningTask(prisma, {taskId, actorUserId, now})` — →APPROVED + approvedBy/At + 게이트 규칙 통과 시 **villa.isSellable=true** + Notification(CLEANING_APPROVED)
     - `rejectCleaningTask(prisma, {taskId, rejectNote 필수, actorUserId})` — →REJECTED + Notification(CLEANING_REJECTED). isSellable 변경 없음(닫힌 채 유지)
     - `createPeriodicCleaningTasks(prisma, now)` — 월 1회 cron: ACTIVE 빌라별 PERIODIC 생성, 같은 달(monthKeyVn) 기존 PERIODIC 있으면 skip(멱등). **isSellable 변경 없음**(SPEC상 게이트 닫기는 체크아웃만)
2. **API** (api-pattern: 첫 줄 401/403, SUPPLIER 스코프 강제)
   - `GET /api/cleaning-tasks` — ADMIN 전체(+status 필터, /inspections·b6용) / SUPPLIER 자기 빌라만 / CLEANER 배정분만. 금액 필드 없음
   - `POST /api/cleaning-tasks/[id]/submit` — photoUrls 제출: ADMIN 또는 해당 빌라 SUPPLIER 또는 배정 CLEANER (타인 403)
   - `POST /api/cleaning-tasks/[id]/approve`·`/reject` — **ADMIN 전용** (reject는 rejectNote 필수)
   - `app/api/cron/periodic-cleaning/route.ts` — CRON_SECRET Bearer (기존 cron 패턴), 월 1회 (Railway 등록은 OPS 잔여)
3. **단위 테스트** — 상태기계 전이 전수, 게이트 규칙, monthKeyVn 경계(연말·UTC/VN 시차)
