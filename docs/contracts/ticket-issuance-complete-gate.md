# 계약: 티켓 발행 완료 시에만 수락 전환 (발주함→예약현황 순서 정정)

- 상태: 착수 (2026-07-12)
- 브랜치: wt/ticket-complete-gate
- 배경(테오): 발주함의 2장짜리 티켓 주문에 1장만 발행해도 즉시 수락 처리되어 예약현황으로
  이동 — 업무 순서가 이상함. **주문 수량을 전부 발행해야 이동**해야 한다.

## 설계 (ADR-0034 개정)

- **자동 수락 게이트 변경**: 벤더 티켓 업로드(POST /api/vendor/orders/[id]/tickets)의
  "발행=수락 겸행" 전이를 **업로드 후 ticketUrls.length ≥ quantity일 때만** 발동
  (VENDOR_ACCEPTED + GUEST 주문 자동 CONFIRMED — 기존 원자 전이·통보 로직 그대로, 시점만 변경).
  미달 업로드는 PENDING_VENDOR 유지 → 발주함에 남고 카운터(1/2)로 진행 표시.
- **유연성 보존(확인시트·FOC 현실 — ADR-0034 취지)**: TICKET+PENDING_VENDOR 카드에서
  **발행 1장 이상이면 수동 "수락" 버튼 노출**(기존 respond accept 재사용) — 확인시트 1장으로
  전원 커버 같은 케이스는 벤더가 명시적으로 수락 가능. 0장이면 기존대로 수락 버튼 숨김(발행 유도).
- 벤더 보드 안내 문구 갱신: "주문 수량만큼 발행하면 자동으로 수락 처리됩니다"(ko/vi).
- 운영자 대리 첨부(상태 불변)·발행 후 삭제(미달돼도 수락 유지)·소비자 부분 발행 경고(PR #254)는 불변.

## 범위
1. `app/api/vendor/orders/[id]/tickets/route.ts` — 전이 조건을 결과 수량 ≥ quantity로
2. `components/vendor/vendor-board.tsx` — 안내 문구·수동 수락 버튼 조건(발행≥1)
3. `messages/ko.json`·`vi.json` — 문구 갱신/추가만
4. 테스트: 미달 업로드=PENDING 유지·통보 미발송 / 충족 업로드=수락+GUEST CONFIRMED+통보 1회 /
   나눠 업로드(1장→1장)=두 번째에 전이 / 수동 수락 경로 회귀

## 수정 금지 구역
