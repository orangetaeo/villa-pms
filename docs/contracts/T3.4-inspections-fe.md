## 범위

`app/(admin)/inspections` — b6 변환 (ADMIN 다크, 좌 목록 + 우 상세 2패널):

1. **좌측 목록**: 청소 태스크 — 빌라명, 유형 배지(체크아웃/정기), 상태 배지(청소 대기/승인 대기/승인됨/반려됨), 제출 시각(Asia/Ho_Chi_Minh 상대·YYYY.MM.DD). **승인 대기(PHOTOS_SUBMITTED) 우선 정렬**. 상태 필터 탭. 데이터는 RSC에서 prisma 직접 조회(T1.5 패턴 — 신규 API 표면 0), 행 선택은 쿼리 파라미터(?task=) 기반
2. **우측 상세 패널**: 선택 태스크의 제출 사진을 **기준 사진(VillaPhoto isBaseline, 같은 빌라)과 나란히** 그리드 — 각 쌍 라벨은 공간명 밝은 라이트 그레이(#E5E7EB — b6 대비 요구). 제출 사진은 photoUrls 순서대로, 기준 사진은 space·sortOrder 순 매칭(개수 불일치 시 남는 쪽 단독 표시). 반려된 태스크는 rejectNote 표시
3. **하단 액션 바**: 승인(초록 — `POST /api/cleaning-tasks/[id]/approve`, 응답 `{task, gateOpened}` — gateOpened=true면 "판매 가능 전환됨" 토스트/배너) / 반려(빨강 — 선택 시 사유 입력란 노출, `POST .../reject` body `{rejectNote}` 필수, 1000자 제한). PHOTOS_SUBMITTED 상태에서만 활성. 409(invalid_transition) 시 안내 + 새로고침
4. 사이드바 "청소 검수" 활성, <768px 반응형(목록→상세 스택 또는 목록 우선 + 상세 진입), adminInspections.* 키(ko/vi)

## 완료 기준 (테스트 가능)
1. ADMIN /inspections: 승인 대기 우선 목록 + 상태 필터, 행 선택 → 상세(제출 vs 기준 사진 쌍) 렌더
2. 승인 → 성공 시 목록 상태 갱신 + gateOpened=true면 판매 가능 전환 안내 노출. 반려 → 사유 필수(공백 거부), 성공 시 REJECTED 반영
3. PHOTOS_SUBMITTED 외 상태에서 승인/반려 버튼 비활성, 409 응답 안내 처리
4. 비ADMIN 차단(페이지 redirect — SUPPLIER·CLEANER도 이 화면 접근 불가, API는 기존 BE 가드), 신규 API 표면 0
5. RSC 직렬화에 고객명·금액 미포함(태스크·사진·빌라명만 — leak-checklist), 영어 잔존 0, 날짜 YYYY.MM.DD
6. b6 충실: 2패널 구조·쌍 라벨 대비(#E5E7EB)·승인 초록/반려 빨강+사유 입력, 사이드바 표준
7. tsc·lint·build 통과, npm test 회귀 0, 충돌 구역 diff 0
