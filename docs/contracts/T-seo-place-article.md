# T-seo-place-article — 푸꾸옥 장소 소개 글 (맛집·카페·쇼핑)

## 배경

현재 블로그 글은 세 종류다: 빌라 글(재고 수만큼) · 서비스 글(9종) · 가이드 글(**고정 8종**).
셋 다 **상한이 있다**. 가이드 8종을 다 쓰면 그 뒤로는 새 글이 나오지 않는다.

맛집·카페·쇼핑 소개는 성격이 다르다 — **테오가 다닐수록 늘어나는, 고갈되지 않는 유일한 글감**이다.
검색 유입 관점에서도 "푸꾸옥 맛집" 계열은 빌라보다 검색량이 훨씬 크고, 그 글에서 빌라로 넘어오게 하는 것이 정석 경로다.

## 이 글 종류만의 위험 (설계가 여기서 갈린다)

1. **AI가 실재하는 가게를 지어낸다.** 빌라·서비스는 우리 DB가 사실 원천이지만, 남의 가게는 아니다.
   존재하지 않는 식당, 틀린 위치, 옮긴 가게를 쓰면 신뢰도 타격 + 구글 저품질 판정.
   → **사실은 사람이 넣고 문장만 AI가 쓴다.** 등록되지 않은 장소는 글에 등장할 수 없다.
2. **변하는 정보를 쓰면 그 순간부터 틀린 글이 된다.** 영업시간·가격·휴무일·전화번호는 수시로 바뀌고
   우리는 갱신할 수단이 없다 → **아예 다루지 않는다**(프롬프트가 아니라 재료 단계에서 배제).
3. **사진 저작권.** 남의 가게 사진을 가져다 쓸 수 없다 → 테오가 직접 찍은 사진만
   (자료 사진 라이브러리 `SeoMedia`에 장소를 연결한다).

## 범위

| # | 항목 | 내용 |
|---|---|---|
| A | 스키마 | `SeoPlace` additive + `SeoMedia.placeId`(장소 사진 연결) |
| B | 운영자 화면 | `/marketing/seo/places` — 장소 등록·수정·비활성, 사진 업로드 시 장소 선택 |
| C | 글 생성 | `lib/seo/place-article.ts` — **카테고리별 묶음 글**(3곳 이상 모이면 1편) |
| D | cron | `seo-draft`에 장소 글 단계 추가 (빌라 → 서비스 → **장소** → 가이드) |
| E | 테스트 | 3곳 미만이면 미생성·같은 장소 재사용 금지·영업시간/가격 재료 부재 |

## 설계 결정

1. **개별 장소 1편이 아니라 묶음 글**(예: "푸꾸옥 카페 3곳 — 직접 가본 곳만").
   카페 한 곳으로 800자를 채우면 지어낼 수밖에 없다. 3곳이면 사실 재료가 3배고 사진도 3장 나온다.
2. **회차가 이어진다.** 새 장소 3곳이 쌓이면 다음 편(`place-cafe-2`, `-3` …). 주제 풀이 고갈되지 않는다.
3. **한 장소는 한 번만 쓴다**(`usedInArticleId`) — 같은 가게가 매 편 반복되면 그 자체가 저품질 신호.
4. **테오의 한 줄 인상(`oneLiner`)은 필수 입력.** 이게 없으면 AI가 채울 수밖에 없다 = 지어내기.
   글의 가치도 여기서 나온다("직접 가본 곳"이 아니면 쓸 이유가 없다).
5. **카테고리는 enum이 아니라 String + 화이트리스트** — 값 추가가 잦을 영역이라 `ALTER TYPE` 비용을 피한다
   (선례: `ContractNegotiation.clauseKey`).
6. 지도 링크(`mapUrl`)는 저장하되 **본문에는 넣지 않는다**(운영자 참고용). 외부 링크는 나중에 정책 잡고 붙인다.

## 재료에서 제외 (틀린 글 방지 — 조회 자체를 안 한다)

영업시간 · 휴무일 · 가격/예산 · 전화번호 · 좌석 수 · 주차 여부 등 **변하는 사실**은 필드 자체를 두지 않는다.
운영자가 굳이 쓰고 싶으면 `oneLiner`/`tips`에 자기 문장으로 넣는다(그건 사람이 책임지는 문장이다).

## 생성 조건

1. 같은 카테고리에서 **아직 글에 안 쓰인 활성 장소 3곳 이상**
2. 각 장소에 `oneLiner` 존재(등록 시 필수) + 재료 합계 **120자 이상**
3. Gemini 성공 + 기존 하한(본문 800자·h2 1개 이상) 통과
4. → 장소가 없으면 아무 일도 일어나지 않는다(현재 0건)

## 완료 기준

1. `/marketing/seo/places`에서 장소 등록 → 사진 업로드 시 그 장소로 연결 가능
2. 같은 카테고리 3곳 미만 → 글 생성 안 함(테스트)
3. 3곳 이상 → `place-<category>-<n>` 초안 1건 + 각 장소 사진 1장씩 본문 배치 + AuditLog(`kind: place`)
4. 이미 쓴 장소는 다음 편에 다시 등장하지 않는다(테스트)
5. 프롬프트·재료에 영업시간·가격·전화 필드가 존재하지 않는다(테스트)
6. `npm run lint && npm run typecheck && npx vitest run tests/seo-place-article.test.ts && npm run build` 통과
