# 계약 — T1.10 공급자 내 빌라 홈(a6) + 빌라 상세 읽기(a10)

2026-06-16 / 담당: UX-VN / 평가: QA / 근거: SPEC F1, design/stitch/a6-my-villas·a10-villa-detail, T1.1(등록 마법사·lib/amenities.ts), 메모리 "Stitch 디자인 우선 규칙"

## 배경
공급자가 마법사로 빌라를 등록(T1.1)했지만, 현재 `/my-villas`는 목록조차 렌더하지 않는 placeholder(제목+FAB만)이고 빌라 상세 화면(`/my-villas/[id]`)은 아예 없다. 등록한 빌라를 볼 수 없는 실질 공백. a6·a10 디자인이 존재한다.

## 수정 금지 구역 (병행 세션)
- T4.1 진행 중: `lib/cleaning.ts`·`lib/hold.ts`·`lib/proposal.ts`(미커밋 — 비접촉)
- 공유 동결: `tailwind.config.ts`, `package.json`, `tsconfig.json`, `app/layout.tsx`, `app/globals.css`, `auth.ts`, `middleware.ts`, **`app/(supplier)/layout.tsx`**(비접촉)
- 기존 lib(amenities·format·serialize·prisma 등)은 import만. `components/supplier/tab-bar.tsx` 사용만. `messages/*.json`은 키 추가만 + 커밋 시 부분 스테이징(정착 패턴)

## 범위

1. **`/my-villas`** (a6 변환 — placeholder 교체, 모바일 라이트 vi):
   - 인사 헤더("Xin chào, {name}!" — 호칭 하드코딩 금지), 제목 "Villa của tôi" + 부제
   - 빌라 카드 세로 리스트(RSC, **supplierId 스코프** — 자기 빌라만): 대표 사진(첫 isBaseline), 상태 배지 오버레이, 빌라명, "N phòng ngủ · M phòng tắm"(영어 beds/baths 금지), 체브론 → 상세
   - **상태 배지 매핑(현존 VillaStatus만)**: ACTIVE+isSellable → "Đang hoạt động"(초록) / ACTIVE+!isSellable → "Chưa thể bán"(빨강 외곽선) + "Xem việc dọn dẹp" 링크(/cleaning) / PENDING_REVIEW·DRAFT → "Chờ duyệt"(주황) / INACTIVE → "Tạm dừng"(회색)
   - "+ Thêm villa" FAB 유지(기존), 빈 상태(빌라 0건) 안내 + 등록 유도
2. **`/my-villas/[id]`** (a10 변환, 신규):
   - RSC, **소유 검증**: villa.supplierId === 세션 아니면 `notFound()`(존재 비노출), 비SUPPLIER redirect
   - 헤더(빌라명 + 상태 배지), 기본 정보 카드("N phòng ngủ · M phòng tắm · Tối đa K khách", 수영장·조식 아이콘), 사진 그리드("N ảnh" + 썸네일, 공간별), 비품 요약 카드("Tiện nghi & đồ dùng" — 카테고리별 개수 "Đồ bếp N · Đồ phòng tắm N · Thiết bị điện N · Minibar N", lib/amenities 카테고리), **원가 카드 "Giá gốc"**(시즌 3종 supplierCostVnd, VND 점 구분 — **supplierCostVnd만 select, salePriceVnd·salePriceKrw·marginValue 절대 미조회·미노출**)
   - 하단 탭바(Villa 활성)
3. **메타데이터**: 두 페이지에 Next.js metadata 추가(T5.5 잔여 — title 누락 해소)
4. i18n: myVillas.* 확장 + villaDetail.* 신규(vi 기본·ko 병기). 비품 라벨은 기존 amenities.* 재사용

## 범위 추가 (2026-06-16 — T1.2b 머지 후 차단 해소)
- **반려(REJECTED) 가시화** — T1.2b가 백엔드(스키마 REJECTED+rejectionReason, 운영자 반려, 편집 라우트 `/my-villas/[id]/edit`+재제출 PUT)를 main 통합(4367911)하며 "공급자 가시화는 T1.10에 인계". 이제 구현 완료:
  - 홈(a6): REJECTED 카드 빨강 배지 "Bị từ chối"(bg-error) + "Sửa và gửi lại" 링크(→ /my-villas/[id]/edit)
  - 상세(a10): 반려 사유 카드(villa.rejectionReason, REJECTED일 때만) + "Sửa và gửi lại" 버튼. 상세 query에 rejectionReason 추가(마진·판매가 select는 여전히 부재 — 누수 불변)
  - i18n: myVillas.status.rejected·myVillas.editResubmit·villaDetail.rejectionTitle·villaDetail.editResubmit (ko/vi) + 키 동등성 테스트 tests/my-villas-i18n-keys.test.ts

## 범위 밖 (여전히 보류)
- 비품 수정(T6.4 잔여 — PATCH API 필요), 요율 편집(공급자는 원가 입력만, 마법사 소관), 사진 추가/삭제
- 운영자 반려 액션·편집 마법사 자체는 T1.2b 소관(완료) — T1.10은 진입 링크만 제공

## 완료 기준 (테스트 가능)
1. SUPPLIER 로그인 → /my-villas: 자기 빌라 카드 목록(사진·이름·침실/욕실·상태 배지), 타인 빌라 미노출. 빈 상태 안내
2. 카드 → /my-villas/[id]: 기본 정보·사진 그리드·비품 요약·원가(시즌 3종) 렌더. 타인 빌라 id 직접 접근 404, 비SUPPLIER redirect
3. **원가만 노출**: salePriceVnd·salePriceKrw·marginValue가 RSC 직렬화·DOM·네트워크에 0건(select 화이트리스트로 구조적 보장 — leak-checklist 핵심)
4. 상태 배지 4종 매핑 정확(ACTIVE 미검수 → Chưa thể bán + 청소 링크)
5. VND 점 구분(공급자 규칙 1.500.000₫), "biệt thự"·영어 잔존 0(vi 화면), 터치 56px+, a6/a10 디자인 충실
6. 메타데이터 title 존재(my-villas·my-villas/[id])
7. tsc(병행 파일 제외)·lint 통과, 충돌 구역 diff 0

## 검증 방법
QA 코드 리뷰 + leak-checklist(특히 VillaRate select에 sale/margin 부재) + lint/tsc + 배포 후 프로덕션 HTTP 증적. 가능하면 제2 공급자 시드로 교차 스코프 실측.
