# T-seo-service-article — 부가서비스 글 자동 생성

## 배경

블로그 초안 cron(`seo-draft`)이 만드는 글은 두 종류뿐이다.

1. **빌라 글** — 공개 자격 빌라 1곳당 1편
2. **가이드 글** — 고정 주제 풀 8종(공항 이동·시즌·아이 동반 …)

앞으로 판매할 **마사지·입장권·BBQ·가이드·차량·조식·오토바이·이발·과일**은 어느 쪽에도 해당하지 않아
**영원히 글이 생성되지 않는다**. 검색 유입이 필요한 상품인데 검색에 걸릴 페이지가 없다.

## 범위

| # | 항목 | 내용 |
|---|---|---|
| A | 주제 정의 | `lib/seo/service-article.ts` — `ServiceType` 9종 ↔ 글 주제(제목·집필 각도) 사전 |
| B | 재료 추출 | 활성 카탈로그 항목에서 **이름·단위·옵션 라벨·설명만** 뽑는다 (가격·원가 제외) |
| C | 생성 | Gemini 본문 + 상품 사진(`photoUrl`) + 자료 사진 라이브러리(서비스 태그) |
| D | cron 연결 | `seo-draft`에 서비스 글 단계 추가 (빌라 글 → **서비스 글** → 가이드 글 순) |
| E | 라이브러리 확장 | 자료 사진 주제 태그에 서비스 9종 추가 (`/marketing/seo/media`) |
| F | 테스트 | 재료에 금액 필드 부재·생성 조건·주제 키 정규화 |

## 절대 규칙 (원칙 2 — 마진 비공개)

- **`costVnd`(매입원가)는 프롬프트·본문·JSON-LD 어디에도 넣지 않는다.**
- **`priceVnd`(판매가)도 공개 글에 넣지 않는다.** 벤더는 자기 원가를 안다 → 공개 판매가 = 마진 역산.
  가격은 상담(`/chat`)·게스트 주문 화면(`/g/[token]`)에서만 노출된다. 글에는 CTA만 둔다.
- 프롬프트에 **금액 자체를 전달하지 않는다** — "쓰지 마라"라고 지시하는 방식은 신뢰하지 않는다.
- 벤더 정보(업체명·연락처·`ServiceVendor`)도 노출 금지 — 직거래 우회 차단.

## 생성 조건 (얇은 콘텐츠 방지)

타입별로 **전부** 충족해야 그 타입 글을 만든다:

1. 그 타입의 **활성 카탈로그 항목 1개 이상**
2. 항목 이름·설명·옵션 라벨을 합친 **재료 글자 수 40자 이상** (이름만 있는 상태로는 쓰지 않는다)
3. 그 타입 글이 아직 없다 (`topicKey = service-<TYPE>`, 상태 무관 — 빌라·가이드와 동일 규칙)
4. Gemini 생성 성공 + 기존 하한(본문 800자, h2 1개 이상) 통과 — 폴백 템플릿 없음

→ **카탈로그가 비면 아무 일도 일어나지 않는다**(현재 0건). 항목을 등록하는 순간부터 대상이 된다.

## 완료 기준

1. 카탈로그 0건 상태에서 `seo-draft` 실행 → 서비스 글 0건, 기존 동작 그대로
2. 활성 항목이 있는 타입 → `service-<TYPE>` 초안 1건 `PENDING_APPROVAL` 생성 + AuditLog(`kind: service`)
3. 재료(facts) 객체에 `price`·`cost`·`vnd`·`krw` 키가 **존재하지 않음**을 테스트로 고정
4. 재료 부족(이름만 있는 항목 1개)이면 생성하지 않고 skip 사유 반환
5. `/marketing/seo/media`에서 서비스 주제로 사진 태그 가능
6. `npm run lint && npm run typecheck && npx vitest run tests/seo-service-article.test.ts && npm run build` 통과
