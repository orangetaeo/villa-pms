# 계약: Zalo zca-js 실연동 (QR 로그인·실송수신) — ADR-0005 이행

날짜: 2026-06-16 / 담당: TDA(설계 선행)→INTEG/BE(구현) / 평가: QA
근거: ADR-0005(OA 폐기·zca-js 채택), reference/nike/src/lib/zalo*.ts(3,498줄 정본), T3.7 온보딩, T6.6 채팅. 점유 선언.

## 배경
현재 lib/zalo.ts는 폐기된 OA API(openapi.zalo.me)로 구현돼 실동작 불가(테오 OA 신청 불가). /messages·/zalo-connect는 UI만 존재. Nike의 zca-js(개인 계정 QR 로그인) 코드를 이식해 실송수신을 활성화한다.

## 1단계 — TDA 설계 (구현 전 필수, 본 계약의 선행 산출물)
docs/decisions/ADR-0006-zalo-runtime.md 작성:
1. **Railway 장기 세션 실행 모델** (최대 관건): zca-js는 WebSocket 리스너를 상시 유지해야 함. Railway 컨테이너(항상 켜진 Node)에서 Next.js instrumentation.ts로 부팅 시 connectAllUsers 호출이 가능한지, 재배포·크래시 시 credential 자동 재로그인 경로, 단일 인스턴스 보장(중복 로그인 밴 위험) 판단. 불가 시 대안(별도 worker 서비스) 제시
2. **스키마 변경**: credential 저장 모델(Nike zalo-credentials 참조 — ZaloAccount/ZaloCredential). 기존 ZaloConversation/ZaloMessage와 통합. 마이그레이션 영향(additive 여부)
3. **기존 코드 교체 계획**: lib/zalo.ts(OA)→zca-js 전송 계층. enqueueNotification/dispatchPendingNotifications(T3.5 OA버전)의 시그니처 유지하며 내부만 교체할지, lib/zalo-pool.ts 신규로 분리할지. 기존 8종 알림·tamtru(T3.6) 호출부 무변경 보장
4. **credential 암호화**: ZALO_CREDS_KEY 환경변수(AES-256-GCM). 환경변수 우선 규칙
5. **단계 분해**: QR 로그인 MVP → credential 영속 → 수신 리스너 → 발송 전환 → 번역 순서

## 2단계 — 구현 (TDA 설계 승인 후)
- zca-js 설치 + lib/zalo-pool.ts·zalo-credentials.ts 이식(Nike 정본 기반, villa-pms 스키마 적응)
- QR 로그인: `app/(admin)/settings` 또는 신규 `/settings/zalo` — startQRLogin → QR base64 렌더 → 스캔 → GotLoginInfo → credential DB 저장 → connected 상태
- ADMIN 연결 상태 표시(a0-zalo-connect 디자인 의미 재사용), 끊김 시 경보
- 수신 메시지 → ZaloConversation/ZaloMessage 저장(b14 /messages 실데이터화)
- 발송 전환: dispatchPendingNotifications가 zca-js 경유(기존 큐·재시도 구조 유지)
