# T-guest-bill-double-count-fix — 게스트 청구 이중 계상(₫+₩ 동시 합산) 수정

- 상태: 완료 (2026-07-13) — QA PASS(결함 0). tsc0·vitest 3011·build 통과. 백필은 배포 후 실행
- 담당: TDA(설계) + BE + FE, QA 검증
- 심각도: **P1 (돈 계산 오류 — 과청구)**
- 배경 (테오 실측 검증 요청): 영수증 총 이용 = VND 2,800,000₫ + KRW 166,000원, 환산 ≈5,695,930₫로 표시.
  실체: 티켓 주문들의 판매가 원천은 `priceVnd`(카탈로그 VND 공시가)이고 `priceKrw`는 **같은 금액의
  주문 시점 환산 표시 스냅샷**(guest·admin 생성 경로 모두 `priceKrw = priceKrwCeil(totalPriceVnd, fx)`,
  카탈로그는 "priceKrw: null — 미사용, 표시 시점 환율 파생" 명시). 그런데 `computeGuestBill`(ADR-0019,
  통화별 단일 컬럼 가정 시절 작성)이 두 컬럼을 **각각 합산** → 사실상 같은 금액을 두 번 청구.
  파급: 체크아웃 청구 카드·승인 하드 게이트 기준액·CheckOutRecord.guestChargeKrw 캐시·게스트 영수증 총액 전부 과대.
  실사례: 해당 예약 실제 청구 2.8M₫인데 5.7M₫로 표시 → 보증금 3M 전액 상계·환불 0 처리(정답: 상계 2.8M + 환불 200,000₫).

## 설계 (TDA 결정)

1. **`computeGuestBill` 원천 통화 1회 계상**: 주문별로 `priceVnd != null` → VND에만 합산(priceKrw 무시 — 표시 스냅샷),
   `priceVnd == null && priceKrw > 0` → KRW에 합산(KRW-원천 주문 보존). 미니바(VND) 불변.
2. **체크아웃 화면(FE)**: 청구 카드 품목 행 = 원천 VND 금액이 주 표기, KRW는 "≈" 참고 표기로 강등(현재는 KRW가 주처럼 보임).
   "₩ 청구 소계(KRW 옵션)" 행은 KRW-원천 주문 합만(현 데이터에선 대부분 0 — 0이면 기존처럼 표시 유지 여부는 FE 재량이되 라벨 혼동 제거).
   가감산·잔여·게이트는 computeGuestBill 결과를 쓰므로 자동 정정.
3. **게스트 영수증**: 총 이용 금액을 record 캐시(guestChargeVnd/Krw) 대신 **로더가 라인에서 라이브 파생**(같은 1회 계상 규칙)
   — 구 레코드(잘못된 캐시)도 즉시 올바르게 표시. 미수납 잔액 계산도 동일 총액 사용.
   서비스 라인 표기도 원천 VND 주 + ≈KRW 참고로 통일.
4. **캐시 백필**: CHECKED_OUT 레코드의 `guestChargeVnd/Krw`를 새 규칙으로 재계산하는 스크립트
   (scripts/backfill-guest-charge.ts, dry-run 기본 + `--apply`) — admin 상세 등 캐시 소비처 정합.
   실행은 배포 후 별도(감사로그 없이 캐시 컬럼만 수정, 수납·보증금 기록은 불변).
5. 스키마 변경 없음.

## 완료 기준
1. 단위테스트: 양컬럼 주문(₫원천+₩스냅샷)=VND 1회만 / KRW-원천(₫ null) 주문=KRW 합산 / 혼합·미니바.
2. 해당 예약 기준 기대값: 총 이용 2,800,000₫·KRW 0원(1m미만 무료 0 제외)·상계 3M → 초과 200,000(영수증 미수납 행 미표시).
3. 체크아웃 화면 총 청구·게이트 기준이 2.8M로 잡히는지(코드 대조·테스트).
4. 영수증 라이브 파생 — 구 레코드 캐시와 무관하게 정정 표시.
5. tsc·vitest 전체·build. QA: 과청구 방향 회귀 없음 + KRW-원천 주문 경로 보존 확인.
