# T4.4 — 베트남 공급자 실사용 테스트 키트 (파일럿)

> **목적**: "베트남 중계인이 **설명 없이** 빌라를 등록하고 캘린더를 관리할 수 있는가?"를 실제 사용자 1명으로 검증한다. (LAUNCH §2 P0/P1 승급 조건, KPI "공급자 자력 등록")
>
> **핵심 규칙 — 테오님(관찰자)은 절대 먼저 도와주지 않는다.** 사용자가 막혀도 30초는 기다린다. 어디서·왜 막히는지가 이 테스트의 전부다. 도와준 순간 데이터가 사라진다.

---

## 1. 합격 기준 (이게 되면 P0→P1 승급)

| # | 기준 | 측정 |
|---|---|---|
| A | 빌라 1채를 **혼자** 끝까지 등록 (사진 포함) | 5단계 마법사 완주 + 제출 성공 화면 도달 |
| B | 캘린더에서 특정 날짜를 **차단/해제** | `/calendar`에서 날짜 토글 1회 이상 성공 |
| C | 치명적 막힘(도움 없이는 못 넘어감) **0건** | 관찰 체크리스트 "🔴 막힘" 0 |
| D | 본인 입으로 "다시 할 수 있겠다" | 사후 설문 Q5 ≥ 4점 |

> A·B·C 모두 통과해야 합격. D는 보조 지표.

---

## 2. 사전 준비 (테스트 전날까지 — 테오님/OPS)

- [ ] **테스트용 SUPPLIER 계정** 발급 (실명 아님, 예: 전화 `0900000200` / 임시 비번). 테스트 후 삭제 대상에 포함
- [ ] 그 계정에 **기존 빌라 0채** 상태 확인 (빈 `/my-villas` "내 빌라가 없습니다" 화면부터 시작해야 자력 등록 검증됨)
- [ ] 사용자에게 줄 **사진 5~8장** 준비 (외관·거실·침실·욕실·수영장 — 사용자 폰 갤러리에 미리 넣어두기). 실제 빌라 사진일 필요 없음, 흐름 검증용
- [ ] 테스트 **기기 = 사용자 본인 폰**(안드로이드/아이폰 무관, 베트남에서 쓰는 그 폰). 데스크톱 금지 — 모바일 우선 UX 검증이 목적
- [ ] 화면 **녹화 켜기** (폰 자체 녹화 or 옆에서 두 번째 폰으로 촬영). 막힌 구간 되돌려 보기용
- [ ] 테오님은 **사용자 뒤/옆**에 앉아 손은 책상 위, 입은 닫는다

---

## 3. 사용자에게 줄 과제 카드 (vi — 그대로 읽어주거나 보여주기)

> 이것 외에 어떤 설명도 하지 않는다. 화면 용어를 풀어주지 않는다.

```
NHIỆM VỤ (3 việc)

1. Đăng nhập bằng số điện thoại và mật khẩu tôi đưa.
2. Tự đăng 1 căn villa (có ảnh) cho đến khi xong.
3. Vào lịch, chặn (block) ngày mai, rồi mở lại.

Cứ làm theo cảm nhận của anh/chị. Tôi không giúp.
Nếu thấy khó ở đâu, cứ nói to ra ("chỗ này khó", "không hiểu").
```

**한국어 참고(테오님용)**
1. 내가 준 전화번호·비밀번호로 로그인
2. 빌라 1채를 사진까지 넣어 **혼자** 끝까지 등록
3. 캘린더에서 **내일** 날짜를 차단했다가 다시 해제

> 안내 화면 `/guide`(가이드 탭)가 있으니, 사용자가 막혔을 때 **스스로 가이드를 찾아가는지**도 관찰 포인트(도와주지 말 것).

---

## 4. 관찰 체크리스트 (ko — 테오님이 실시간 기록)

각 단계: ⏱️ 소요 시간 + 🟢 무리없음 / 🟡 잠시 멈칫 / 🔴 막힘(도움 없이 못 넘김) + 한 줄 메모(사용자가 한 말·표정)

### 4.1 로그인
| 항목 | 판정 | 시간 | 메모 |
|---|---|---|---|
| 전화번호 입력 형식 이해 (하이픈/공백) | | | |
| 로그인 성공 → 홈 도달 | | | |

### 4.2 빌라 등록 마법사 (`/my-villas/new`, 5단계)
| 단계 | 화면 | 판정 | 시간 | 메모 |
|---|---|---|---|---|
| 진입 | 빈 목록에서 "+ 빌라 추가" 버튼을 스스로 찾는가 | | | |
| 1/5 기본정보 | 빌라명·인원·침실/욕실 수·수영장·조식 토글 | | | |
| 2/5 위치 | 입력 or "건너뛰기"를 판단하는가 | | | |
| 3/5 사진 | 슬롯별 사진 업로드(외관·거실·침실N·욕실N…) — **가장 막히기 쉬운 곳** | | | |
| 4/5 비품 | 탭 4종 + 미니바 수량 스테퍼 | | | |
| 5/5 원가 | 시즌 3카드 숫자 입력 + 제출 버튼 활성화 | | | |
| 완료 | 제출 성공 화면 / 목록에 새 빌라(검수 대기 배지) 노출 | | | |

> **사진 단계 집중 관찰**: 슬롯 개념(외관/거실/침실1/침실2…)을 이해하는가? 한 슬롯에 여러 장 vs 슬롯마다 1장을 헷갈리는가? 업로드 진행/완료를 인지하는가?

### 4.3 캘린더 (`/calendar`)
| 항목 | 판정 | 시간 | 메모 |
|---|---|---|---|
| 캘린더 탭/메뉴를 스스로 찾는가 | | | |
| 내일 날짜를 눌러 차단 상태로 바꾸는가 | | | |
| 차단을 다시 해제하는가 | | | |
| 셀 색/패턴(공실·차단·예약)의 의미를 이해하는가 | | | |

### 4.4 전반 관찰 (자유 기록)
- 가장 오래 멈춘 화면: ______
- 도와주고 싶었지만 참은 순간(=실제 UX 결함 후보): ______
- 사용자가 소리내어 말한 불만/혼란 (원문 vi 그대로): ______
- 가이드(`/guide`)를 스스로 열어봤는가: ______

---

## 5. 사후 설문 (5문항 — 끝나고 바로, vi로 묻고 ko로 기록)

1~5점 (1=전혀 아니다, 5=매우 그렇다)

| # | vi | ko | 점수 |
|---|---|---|---|
| Q1 | Việc đăng villa có dễ không? | 빌라 등록이 쉬웠나요? | |
| Q2 | Có chỗ nào anh/chị thấy bối rối không? (chỗ nào?) | 헷갈린 곳이 있었나요? (어디?) | |
| Q3 | Bước tải ảnh có rõ ràng không? | 사진 올리는 단계가 명확했나요? | |
| Q4 | Anh/chị có cần tôi giải thích không? | 제 설명이 필요했나요? (낮을수록 좋음) | |
| Q5 | Lần sau anh/chị tự làm một mình được chứ? | 다음엔 혼자 할 수 있겠어요? | |

> Q2의 자유 답변(원문)이 가장 중요. 그대로 받아 적는다.

## 6. 결과 판정 & 후속

테스트 종료 후 테오님이 채우는 요약:

```
일시:            / 사용자(역할·나이대):
빌라 등록 완주:  ✅ / ❌ (막힌 단계: ___)
캘린더 토글:     ✅ / ❌
🔴 막힘 건수:    ___ 건 (목록: ___)
설문 Q5 점수:    ___ / 5
판정:           합격(P1 승급) / 불합격(재설계 후 재테스트)
```

**후속 처리 규칙**
- 🔴 막힘 / 🟡 멈칫 → 전부 **IDEAS.md에 먼저 기록** (LAUNCH §4 개선 루프). 즉시 코드 수정 금지
- 합격이면 LAUNCH §1 체크리스트 "베트남 사용자 1명 실사용 테스트 통과 [ ]" → [x], TASKS T4.4 완료 처리
- 불합격이면 막힌 단계의 화면 결함을 UX-VN 태스크로 승격(우선순위 회의)
- 발견 교훈은 `.claude/skills/ux-vn/vn-ux-principles.md` "교훈 축적"에 패턴화

## 7. 비고

- 이 키트는 **P0 파일럿(테오 동행)** 용. P1 단계에서는 같은 카드를 **원격으로** 보내 동행 없이 재현되는지 확인한다.
- Zalo 알림(`/zalo-connect`) 흐름은 이 테스트 범위 밖 — Zalo 라이브 활성화 후 별도 검증.
- 사진은 실제 빌라 불요(흐름 검증). 단, 합격 후 실데이터로 1채는 진짜 사진으로 재등록 권장.
