# 실사용 테스트 관찰 킷 — 전산 초보가 설명 없이 쓸 수 있는가 (T4.4)

테오가 실제 사용자(베트남 공급자·청소 담당 등)를 만났을 때 **10~20분**으로 진행하는 관찰 테스트.
목적: "튜토리얼·화면 안내가 충분한가"를 추측이 아니라 관찰로 판정하고, 막힌 지점을 다음 개선 목록으로 만든다.

## 진행 규칙 (가장 중요)

1. **개입 금지** — 과제를 말해준 뒤엔 조용히 지켜본다. 물어봐도 "화면에 있는 걸로 해보세요"로만 답한다.
   (한 번 도와주면 그 지점의 데이터가 사라진다 — 도와준 순간이 곧 "막힌 지점"으로 기록 대상)
2. **60초 규칙** — 같은 화면에서 60초 이상 헤매거나 포기 의사를 보이면 그 지점을 기록하고 다음 과제로.
3. 사용자 본인 폰으로, 데모 계정으로 진행 (실데이터 오염 방지).
4. 시작 전 한마디: "이건 당신 시험이 아니라 프로그램 시험입니다. 어려우면 프로그램 잘못이에요."

## 역할별 과제 시나리오

과제는 **결과만 말한다** ("어떻게"는 절대 설명하지 않는다).

### A. 공급자 (핵심 — 최우선 테스트 대상)
| # | 과제 (사용자에게 이렇게만 말한다) | 성공 기준 |
|---|---|---|
| A1 | "이 주소로 들어가서 로그인해 보세요" | 로그인 화면 도달 + 로그인 성공 |
| A2 | "빌라를 하나 등록해 보세요" (사진 몇 장 미리 폰에 준비) | 마법사 6단계 완주 → 등록 완료 화면 |
| A3 | "다음 주 화·수요일은 방을 팔지 않게 막아 보세요" | 캘린더에서 해당 날짜 차단 |
| A4 | "청소가 끝났다고 사진을 올려 보세요" | 청소 태스크 진입 → 사진 제출 |
| A5 | "방금 본 화면 안내를 다시 볼 수 있나요?" | "?" 버튼을 스스로 찾는가 |

### B. 청소 담당
| # | 과제 | 성공 기준 |
|---|---|---|
| B1 | 로그인 → "오늘 할 일이 있는지 확인해 보세요" | 태스크 목록·날짜 칩 사용 |
| B2 | "청소 끝난 사진을 올려 보세요" | 슬롯별 업로드 → 제출 |

### C. 파트너 (여행사·랜드사 — 한국인)
| # | 과제 | 성공 기준 |
|---|---|---|
| C1 | "받은 제안서를 열어 예약까지 해보세요" | 제안 탭 → 열람·예약 링크 → 가예약 |
| C2 | "입금했다고 알려 보세요" | 미수 탭 → 입금통보 |

### D. 벤더 (부가서비스)
| # | 과제 | 성공 기준 |
|---|---|---|
| D1 | 로그인 → "들어온 주문을 받아 보세요" | 발주함에서 수락 |
| D2 | "일을 끝냈다고 알려 보세요" | 일정 탭 → 완료보고 |

## 기록 표 (과제마다 한 줄)

| 과제 | 결과(○ 혼자 완주 / △ 헤맴 후 완주 / ✕ 포기·개입) | 막힌 화면 | 그때 사용자가 한 행동·말 | 튜토리얼을 봤나(스킵했나) |
|---|---|---|---|---|
| A1 | | | | |
| A2 | | | | |
| … | | | | |

**관찰 포인트 (표 못 채워도 이것만은):**
- 첫 코치마크가 떴을 때 반응 — 읽는가, 반사적으로 건너뛰는가
- 뭘 눌러야 할지 몰라 **화면을 위아래로 반복 스크롤**하는 순간 (= 안내 공백 신호)
- 사용자가 소리 내어 한 질문 그대로 받아적기 ("이 숫자가 뭐예요?" 같은 것 — 문구 개선 원문이 됨)

## 판정 기준

- **합격**: 핵심 과제(A2·A4 / B2 / C1 / D1)를 ○ 또는 △로 완주
- **개선 필요**: ✕가 하나라도 나온 과제 — 막힌 화면·행동 기록이 곧 개선 티켓
- △가 많으면 해당 화면의 안내 문구·배치 재검토 (완주했어도 헤맨 시간이 이탈 위험)

## 결과 → 개선으로 잇는 법

1. 기록 표를 세션에 그대로 붙여넣고 "실사용 테스트 결과로 개선 태스크 만들어줘"라고 지시
2. 막힌 지점별로: 문구 수정(즉시) / 인라인 안내 추가(소) / 화면 구조 변경(중 — 회의 필요)으로 분류됨
3. 같은 지점에서 2명 이상 막히면 우선순위 최상
