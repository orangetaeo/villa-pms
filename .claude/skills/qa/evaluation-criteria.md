# Skill: 평가 기준 및 스프린트 계약 (Anthropic harness design 2026.3 적용)

## 원칙: 작업자-평가자 분리
- 코드를 작성한 에이전트는 자기 작업을 "통과"로 판정할 수 없다 (자기평가는 관대해지는 경향)
- QA는 의도적으로 회의적인 평가자다 — 칭찬이 아니라 결함을 찾는 게 일

## 스프린트 계약 (작업 시작 전 필수)
코딩 전에 담당 에이전트가 계약 초안을 작성하고 QA가 검토·합의한다:
- 파일: docs/contracts/<태스크번호>.md
- 내용: ① 구현 범위 ② 테스트 가능한 완료 기준 목록(구체적 행동 단위) ③ 검증 방법
- 합의 전 코딩 금지. 완료 기준이 모호하면 QA가 반려

## 실사용 검증 (코드 리뷰만으로 통과 금지)
- QA는 Playwright MCP로 실행 중인 앱을 실제 사용자처럼 클릭하며 검증
- UI 동작 + API 응답 + DB 상태 3개 층 모두 확인
- 계약의 완료 기준을 하나씩 실행하며 통과/실패 기록

## 채점 기준 + 하드 임계치 (하나라도 미달 = 스프린트 실패 → 재작업)
| 기준 | 내용 | 임계치 |
|---|---|---|
| 기능성 | 계약 완료 기준 통과율 | 100% |
| 권한 안전 | 누수 4종 (leak-checklist.md) | 0건 |
| 제품 깊이 | 빈 상태·에러·로딩 처리 존재 | 필수 화면 전부 |
| 디자인 품질 | DESIGN.md 4기준 (UI 작업 시) | 각 기준 "양호" 이상 |

## 실패 시
- 구체적 결함 목록(재현 절차 포함)을 담당 에이전트에 전달 — "다시 해" 금지, 행동 가능한 피드백만
